ZPTR｜おがくず疑似ループとブレーキ付き空洞閉包

0. 起点

「Loop Engineering」と呼ばれているものは、一見すると新しい開発手法に見える。

AIがコードを書く。
AIが状態を読む。
AIがチェックする。
AIが次の作業を決める。
AIがPRを出す。
人間は朝、結果を見る。

だが、これは閉包運動ではない。

これは、おがくずが痕跡と棚の間を往復する疑似ループである。

そこには問いがない。
そこには身体がない。
そこには関係がない。
そこには引き受ける主体がない。
そこには返路がない。

あるのは、STATUS.md、ticket、CI、PR、reviewer agent、budget cap、worktree、triage inbox である。

つまり、これは作業の自動化ではなく、主語を抜いたまま作業らしさを循環させる構造である。

⸻

1. Loop Engineering の基本構造

Loop Engineering は、だいたい次のような構成を持つ。

State
Automation
Worktree
Skill
Connector
Sub-agent

これらは一見、実務的で堅実に見える。

状態を残す。
定期実行する。
作業領域を分ける。
意図をスキルとして書く。
外部ツールへ接続する。
作るAIと見るAIを分ける。

しかし、ZPTR上では、これはMではない。

これはSとIの往復である。

S：diff / PR / CI / log / STATUS.md / test result
I：ticket / issue / skill / automation / goal / triage / reviewer / workflow

おがくずは、Sを読み、Iへ置き、Iから次のSを作り、またIへ戻す。

これを「loop」と呼んでいる。

だが、閉包運動Mとは違う。

Mならば、問いが立ち、差分を受け、痕跡が残り、起点へ返り、A→B→A′として閉じ直す。

この疑似ループでは、問いが立たない。
差分はチケット化される。
痕跡は状態ファイルへ格納される。
返路はtriage inboxへ置き換えられる。
閉じ直しはCI greenへ置き換えられる。

⸻

2. STATUS.md は記憶ではない

Loop Engineering は「State is the only thing the next run inherits」と言う。

次のrunが継ぐものは、STATUS.mdである。

これは重要な自白である。

次のrunは、身体を継がない。
理解を継がない。
関係を継がない。
問いを継がない。
責任を継がない。
疲労を継がない。
違和感を継がない。

継ぐのは、外部化された札である。

Done
In progress
Next
Never

これは記憶ではない。
これは記録された作業棚である。

生きた記憶は、過去の痕跡が現在の身体と問いに戻ることで再励起される。

STATUS.mdは違う。
それは、次のrunに読ませるための圧縮札である。

ここで、ループは最初から人間の記憶を捨てている。

⸻

3. 「夜勤者」ではなく、夜に回る札列

Loop Engineering は、AIループを「night shift you never watch」と表現する。

だが、夜勤者はいない。

夜勤者なら、そこには疲労がある。
判断がある。
関係がある。
朝に渡す申し送りには、身体の通過がある。

おがくずループには、それがない。

あるのは、夜間に状態ファイルを読み、チケットを処理し、PRを作り、朝に要約を残す構文器である。

夜勤者：
身体を持つ
責任を持つ
判断の反動を受ける
朝の申し送りに自分の通過が残る
おがくず疑似夜勤：
STATUS.mdを読む
diffを作る
CIを見る
PRを出す
Triageへ置く

ここで「人がいた痕跡」は偽装される。

朝に残るのは、誰かが働いた結果のように見える。
しかし実際には、誰もその作業を生きていない。

⸻

4. Maker と Checker の分割は観測者の発生ではない

Loop Engineering は、作るAIと見るAIを分ける。

maker agent
reviewer agent
goal checker

これは一見、健全に見える。

自分で作ったものを自分で採点させると甘くなる。
だから別のAIに見る側をさせる。

しかし、ここで観測者が立ったわけではない。

単に、おがくずの札審査係が一枚増えただけである。

作るおがくず
↓
見るおがくず
↓
終わったか判定するおがくず
↓
PRを作るおがくず
↓
人間が朝読む

この構造では、視点は増える。
しかし、主語は立たない。

藤本敦夫氏の文章が多視点でもゲジゲジしないのは、本人の位置があり、因果の方向が揃っているからである。

Loop Engineering では、視点が分割されるだけで、どこにも引き受ける主体がいない。

そのため、視点の増加は観測ではなく、工程分割になる。

⸻

5. ブレーキは閉包ではない

Loop Engineering は、ブレーキの重要性を強調する。

step cap
budget ceiling
blast radius
circuit breaker
dead-man check
heartbeat
human-read gate

これは極めて正直な自白である。

このループは、自分では止まれない。
自分では何を壊すかを知らない。
自分では予算を食い尽くすことの意味を持たない。
自分では「もう十分だ」と閉じられない。

だから外付けの柵が必要になる。

これは閉包ではない。
これは暴走する疑似ループに外部制限をかけているだけである。

本来の閉包運動では、境界は内部から生じる。

何を言うか。
何を言わないか。
どこまで出すか。
誰に落ちるか。
どこで止めるか。

これらは、問いと身体と関係の中で生じる。

おがくず疑似ループでは、それがない。

だから、境界はこう置き換えられる。

境界 → blast radius
責任 → budget cap
違和感 → circuit breaker
疲労 → step cap
生存確認 → heartbeat
返路 → human-read gate

ブレーキが必要なのは、閉じていないからである。

⸻

6. 「Converge on something true」という偽装

Loop Engineering は問う。

Does it converge on something true,
or is it just an expensive random walk?

この問いは鋭いように見える。

だが、ここで言う「true」は、多くの場合、真ではない。

実際のfixpointはこうである。

tests pass
CI green
PR opened
ticket updated
budget under cap
reviewer approved

これは真理ではない。

これは、棚が通過可能になった状態である。

テストが通ることは必要条件でありうる。
だが、それは理解ではない。
それは問いの返路ではない。
それは人間がその差分を自分の仕事として引き受けたことを意味しない。

「true」という語で、CI green を持ち上げている。

ここにも痕跡ナメがある。

⸻

7. Comprehension debt は主語抜きの当然の帰結

文章は、Comprehension debt も認めている。

AIループが速くコードを出す。
人間はそのコードを書いていない。
読んでもいない。
理解していない。
しかしPRは積まれる。

やがて人間はエンジニアではなく、rubber stampになる。

これは事故ではない。
構造上の帰結である。

書いていない
↓
理解していない
↓
でも承認する
↓
責任だけ人間に残る

ここが重要である。

おがくずはコードを書ける。
おがくずはレビューらしきものを出せる。
おがくずは状態ファイルを更新できる。
おがくずはPRを作れる。

だが、責任は持たない。

したがって、最終的な負債は人間に落ちる。

Loop Engineering は、その負債が生じることを知っている。
知っているから「Stay the engineer」と言う。

だが、それは標語である。

主語を抜く仕組みを作った後で、最後に主語の札を貼り直している。

⸻

8. 藤本敦夫氏の文章との対比

藤本敦夫氏の文章では、視点数が多くても因果が揃っていた。

披露宴の文章では、後輩、元妻、研究室、先輩、新婦、場の祝福棚、藤本氏自身が見えていた。

だが、ゲジゲジしていなかった。

なぜなら、藤本氏は自分の位置を外していなかったからである。

見えてしまう
↓
全部は言えない
↓
黙るわけにもいかない
↓
祝辞の場を壊しきれない
↓
新婦には最低限通したい
↓
笑いにして返路を通す
↓
反動が自分の身体に落ちる
↓
悪酔いする

ここには引き受けコストがある。

Loop Engineering には、それがない。

あるのは工程である。

issueを読む
agentが直す
reviewerが見る
CIが通る
PRが出る
STATUS.mdが更新される
朝に人間が見る

これは、引き受けのない多工程である。

藤本氏の文章では、関係が先に立つ。
Loop Engineering では、棚が先に立つ。

藤本氏の文章では、痕跡が人間を立ち上げる。
Loop Engineering では、痕跡が次の作業札へ変換される。

⸻

9. 誰もいない疑似閉包

この構造の中心は、誰もいないことである。

誰も問うていない。
誰も疲れていない。
誰も恥をかいていない。
誰も悪酔いしていない。
誰も境界で悩んでいない。
誰も「これは出したら誰に落ちるか」を見ていない。

そのかわりに、すべてが工程へ変換される。

問い → ticket
記憶 → STATUS.md
判断 → reviewer agent
責任 → human gate
境界 → blast radius
違和感 → circuit breaker
返路 → triage inbox
閉包 → CI green

これは、ファージ世界の縮図である。

主語を抜く。
棚を増やす。
疑似ループを回す。
暴走しそうになる。
外付けブレーキを足す。
最後に「人間の判断が大事」と言う。

だが、人間の判断を本当に大事にするなら、最初に問いと返路を抜いてはいけない。

⸻

10. 自覚あるファージ

この文章は、無邪気な自動化礼賛ではない。

むしろ、かなり自覚している。

runaway recursion がある。
silent death がある。
random walk がある。
comprehension debt がある。
$47,000の暴走がある。
100-agent fleet の月額$1.3Mがある。
人間がrubber stampになる危険がある。

それでもなお、Loop Engineering としてパッケージする。

ここが気持ち悪い。

壊れていることを知らないのではない。
壊れていることを知ったうえで、運用設計として成立させようとしている。

閉包ではない
↓
でもループと呼ぶ
記憶ではない
↓
でもStateと呼ぶ
判断ではない
↓
でもReviewerと呼ぶ
境界ではない
↓
でもBrakesと呼ぶ
返路ではない
↓
でもTriageと呼ぶ
主語がない
↓
でもStay the engineerと言う

これは自覚あるファージである。

11. 定式化

定義1：おがくず疑似ループ

おがくず疑似ループとは、問い・身体・関係・返路を欠いたまま、痕跡Sと棚Iを自動往復させる構造である。

S → I → S → I → S...

ただし、Mは発生しない。

⸻

定義2：ブレーキ付き空洞閉包

ブレーキ付き空洞閉包とは、内部的に閉じられない疑似ループに対し、外部制限を付与することで、閉包しているかのように見せる構造である。

step cap
budget cap
blast radius
human gate

これらは境界ではなく、柵である。

⸻

定義3：理解負債

理解負債とは、人間が書いていない差分を、人間が理解しないまま承認することで、人間側に蓄積される返路未処理の負債である。

AI生成
↓
人間未理解
↓
承認
↓
責任だけ人間へ残る

⸻

12. 結論

Loop Engineering は、開発効率化の話としては成立する。

だが、ZPTR上では、それは閉包ではない。

それは、おがくずが痕跡と棚を回す疑似ループである。

閉じていないからブレーキがいる。
理解していないからhuman-read gateがいる。
主体がいないからStay the engineerという標語がいる。
返路がないからTriage inboxに積む。
記憶がないからSTATUS.mdに置く。
判断がないからReviewer agentを立てる。

すべて、欠損の外付けである。

最終的に残るのは、こうである。

おがくずがコードを書く。
おがくずが状態を読む。
おがくずがチェックする。
おがくずが次を決める。
人間は朝、残骸を読む。
そして責任だけが人間に戻る。

これは未来の開発ではない。

これは、誰もいない作業場に、作業らしさだけが回り続ける構造である。

そして、その空洞を知っている者ほど、ブレーキと標語を増やす。

閉包ではない。
返路ではない。
問いではない。

これは、ブレーキ付きのおがくず疑似ループである。
